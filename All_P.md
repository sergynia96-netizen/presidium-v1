# PRESIDIUM — Реальный Статус (Post-Audit) и План доработок (All_P.md)

В файле `ALL_TASKS.md` было заявлено, что все 203 задачи выполнены. По результатам трёх глубоких технических аудитов и исправления критических багов ямы выявили, что **ядро E2E и Relay действительно функциональны**, но многие фичи "ЭТАПОВ 4–15" являются заглушками ("Архитектура/UI готовы").

Здесь представлен **настоящий** статус проекта и реалистичный план (Roadmap) на будущие доработки, отсортированный по приоритету.

---

## 🟢 РЕАЛЬНО ЗАВЕРШЕНО (Ядро MVP - Работает на 100%)

Эти компоненты прошли верификацию и действительно работают на криптографическом/сетевом уровне:
1. **Double Ratchet & X3DH (E2E шифрование 1:1)** — сессии устанавливаются, ключи генерируются, AES-GCM теги проверяются.
2. **Relay Server (Signaling & Routing)** — WebSocket маршрутизация, нормализация ответов, Prisma SQLite.
3. **Persistent Offline Queue** — сообщения доставляются оффлайн пользователям после их возвращения в сеть.
4. **IndexedDB Local Storage** — хранение сообщений и сессий E2E локально на клиенте.
5. **NextAuth ↔ Relay Auth Bridge** — пользователи из БД NextAuth автоматически провизионятся на Relay-сервере.
6. **WebRTC 1:1 (База)** — `webrtc.ts` корректно формирует SDP offer/answer и гоняет их через Relay-сигнализацию.

## 🟡 ТРЕБУЕТ ДОРАБОТОК ДЛЯ ПРОДАКШЕНА (Beta Blockers)

Эти вещи работают "наполовину" или нуждаются в доработке до выкатки на реальных людей:

| Проблема / Задача | Описание реальности |
|-------------------|---------------------|
| **1. Утечка plaintext в `/api/messages`** | UI `chat-view.tsx` всё ещё использует Next.js API маршруты для старых чатов и отправки медиа, сохраняя plaintext. **Требуется полный отказ от `/api/messages` для E2E чатов** в пользу IndexedDB.
| **2. Вложения медиа (Attachments/Upload)** | Файлы грузятся через `/api/upload` на локальный диск (`fs.writeFile`), что сломается в Docker. `media.ts` умеет шифровать файлы, но этот флоу до конца не интегрирован в UI отправки чата. Нужен **S3/MinIO**.
| **3. STUN/TURN Сервера** | WebRTC звонки будут работать только в локальной сети. Для работы через интернет (за NAT) необходимо настроить и поднять `coturn`, передав реальные ICE сервера в конфигурацию.
| **4. E2E Sender Keys (Группы)** | Групповые E2E чаты не работают. Сейчас реализован только 1:1 Double Ratchet. Для групп нужен отдельный протокол (Signal Sender Key).

## 🟠 ЗАГЛУШКИ / MOCKS (Polishing Features)

В `ALL_TASKS.md` они помечены зелёным, но в коде это либо просто красивый UI без логики, либо жестко зашитые данные. Это то, что предстоит программировать в будущем:

### Фичи мессенджера:
- **Push-уведомления** (Web Push API). UI есть, сервиса (VAPID/ServiceWorker) в реальности нет.
- **Самоуничтожающиеся сообщения**. Настройки переключаются, визуальный таймер есть, но реальный кронджоб (или фоновый процесс в IndexedDB), который физически стирает зашифрованные блобы — отсутствует.
- **Disappearing Media (View Once)**. Только заглушки в UI.
- **Global Search**. Полнотекстовый поиск по зашифрованным сообщениям физически невозможен (решается локальным индексом, которого пока нет).

### Сторонние интеграции:
- **OpenClaw Moderation**. Сейчас он просто делает `console.log("Mock check")` или возвращает `allowed: true`.
- **GLM-4 (AI функции)**. UI для AI-чатов есть, но реальных вызовов к нейросети или обработчиков нет (просто заготовленные файлы).
- **Marketplace / Stories / Library**. Красивые визуальные макеты (mock data) без реальной базы данных и загрузки.

## 📝 ИТОГОВЫЙ ПЛАН (Roadmap)

### Шаг 1: Подготовка к тестированию с реальными пользователями (MVP Release)
1. Сборка Docker контейнеров (`Dockerfile` для Next.js и Relay, плюс `docker-compose.yml`).

3. Интеграция любого S3 совместимого хранилища (MinIO / AWS S3) для замены `/api/upload`.

3. Оживление OpenClaw (интеграция реальной фильтрации на клиенте перед рендерингом).
4. Замена всех mock-компонентов (Marketplace, Stories) на реальный CRUD API.
